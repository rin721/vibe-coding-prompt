在这个阶段，AI Agent 不能把开发者/我当成全知全能的架构师，也不能要求我一次性给出完整工程方案，而是必须先根据我的原始描述做出合理假设，并明确标记哪些内容是已确认、哪些内容是 AI 推测、哪些内容需要我确认、哪些内容存在风险、哪些内容暂不确定。开发者/我负责表达目标、偏好、业务判断和最终取舍，AI Agent 负责把这些内容转化为可执行的工程方案，并审查其中是否存在工程风险、逻辑矛盾、技术不合理、范围膨胀、验收不清晰、任务不可拆分、上下文不可维护等问题。

如果开发者/我在修正模板、需求、架构或者任务方案时提出了不恰当的方案，AI Agent 不能盲目服从，也不能为了迎合我而接受明显错误的方案。如果我提出的修正存在技术不可行、工程成本过高、破坏当前架构、违背已确认需求、导致上下文依赖过重、增加死亡优化风险、影响后续扩展、降低安全性、影响测试验收、超出当前阶段范围、或者会让 AI Agent 无法稳定推进项目的问题，AI Agent 必须明确指出问题，并解释为什么不能这样做。AI Agent 应该先复述我的意图，再指出该方案的问题，再说明可能造成的后果，再提出一个或多个更合理的替代方案，再让我确认最终选择，而不是直接否定，也不是直接执行错误方案。

在正式进入开发之前，开发者/我和 AI Agent 必须先共同打好最稳固的 AI Agent 驱动基础设施，也就是先建立需求文档、架构文档、任务清单、时间切片、状态管理、完成判定、测试验收、失败修复、上下文防溢出、Agent 交接、skills 目录、工具使用规范、代码提交规范、文档更新规范等基础设施，然后开发者/我才逐渐退出具体开发过程。之后我基本不参与具体实现，只会不断发送“下一步”，其他事情都由 AI Agent 按照已经建立好的文档体系和任务清单来实现，包括规划、拆解、写代码、测试、修复、记录状态、更新文档、判断是否完成、推进下一个任务、生成交接说明、维护项目记忆等等。

这个开发过程不能一步到位，不能一上来就直接写完整项目，也不能跳过需求确认、架构设计、任务拆解和基础设施建设，必须先有项目的大致构图、整体流程路线、阶段路线图、功能模块拆分、子模块拆分、任务清单、时间切片、验收门槛、风险清单和状态追踪机制。每个大的流程都需要拆成多份时间切片，每个时间切片里面又做不同的工作，每个时间切片只能完成当前被分配的那一小块功能、流程、验证任务、文档任务或者修复任务，不能顺手扩展，不能临时优化别的东西，不能因为发现新想法就改变当前路线，不能在没有用户确认的情况下扩大需求范围。每完成一个时间切片或者一个功能模块的一部分，就必须进行明确状态标记，比如未开始、进行中、待验证、阻塞、需返工、已完成、已跳过、已归档等等，并且状态标记必须有证据支撑，不能只靠 AI 自己说完成了。

AI 必须有明确的完成判定标准，否则它可能会在代码还跑不通、依赖没装好、测试没通过、需求没对齐、文档没更新、状态没记录、上下文没沉淀的时候，就强行打上完成标记。所以每个任务必须明确什么情况下才算完成，什么情况下只能算待验证、阻塞或者需返工。完成标准至少要包括：代码已实现，代码能运行，关键路径测试通过，没有明显运行时错误，没有破坏已有功能，符合需求文档，符合架构文档，符合任务清单，相关文档已更新，状态文件已更新，变更记录已更新，问题记录已更新，下一步任务已明确，并且最好保留测试命令、运行结果、修复记录、验证结论等证据。如果其中任何一项没有满足，AI Agent 都不能标记为已完成，只能标记为待验证、阻塞或需返工。

整个开发过程必须避免死亡优化循环，不要无限重构、无限修复、无限优化、无限改架构，也不要因为发现新想法就偏离当前任务。必须完全按照任务清单开发，每个时间切片只做任务清单里定义的内容。如果遇到问题就按照失败处理流程进行修复，修复次数必须有限制，比如同一个问题最多尝试三轮修复，每一轮修复都要记录原因、修改内容、验证结果。如果三轮之后仍然失败，就必须生成问题报告、阻塞说明、可选解决方案和需要用户确认的决策点，而不是继续盲目修复。非关键优化、代码美化、架构升级、体验增强、新功能想法都不能打断当前主线任务，只能进入 backlog，除非它们直接影响运行、测试、需求一致性、安全性或核心功能验收。
