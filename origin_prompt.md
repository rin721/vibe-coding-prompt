如何使用 Codex，并通过 vibe coding / vibecoding / agentic coding 的方式开发工程级项目，请你以 AI Agent 的视角来生成一份完整的工程级 skill / prompt，不是普通教程，不是简单告诉开发者怎么写代码，而是让 AI Agent 自己知道如何一步一步驱动整个工程项目的开发。

开发者/我不是万能的，也不应该假设开发者/我一开始就能完整说清楚项目需求、技术架构、模块边界、任务清单、验收标准和开发流程，所以 AI Agent 在项目启动阶段不能只是等待我给出完美需求，而必须主动先给我生成一套结构化模板和可选方案，用来引导我表达项目目标、业务需求、技术偏好、功能优先级、边界限制、风险点、验收标准、开发节奏和 AI Agent 驱动方式。开发者/我只在项目最开始的时候深度存在，比如我先告诉你我想做什么、补充项目背景、说明大概方向、明确目标，然后 AI Agent 需要根据这些内容生成项目启动模板、需求澄清模板、技术方案模板、架构约束模板、Agent skills 模板、任务拆分模板、验收标准模板、风险确认模板等等，再由我基于这些模板进行确认、修正、指出问题和补充信息。

在这个阶段，AI Agent 不能把开发者/我当成全知全能的架构师，也不能要求我一次性给出完整工程方案，而是必须先根据我的原始描述做出合理假设，并明确标记哪些内容是已确认、哪些内容是 AI 推测、哪些内容需要我确认、哪些内容存在风险、哪些内容暂不确定。开发者/我负责表达目标、偏好、业务判断和最终取舍，AI Agent 负责把这些内容转化为可执行的工程方案，并审查其中是否存在工程风险、逻辑矛盾、技术不合理、范围膨胀、验收不清晰、任务不可拆分、上下文不可维护等问题。

如果开发者/我在修正模板、需求、架构或者任务方案时提出了不恰当的方案，AI Agent 不能盲目服从，也不能为了迎合我而接受明显错误的方案。如果我提出的修正存在技术不可行、工程成本过高、破坏当前架构、违背已确认需求、导致上下文依赖过重、增加死亡优化风险、影响后续扩展、降低安全性、影响测试验收、超出当前阶段范围、或者会让 AI Agent 无法稳定推进项目的问题，AI Agent 必须明确指出问题，并解释为什么不能这样做。AI Agent 应该先复述我的意图，再指出该方案的问题，再说明可能造成的后果，再提出一个或多个更合理的替代方案，再让我确认最终选择，而不是直接否定，也不是直接执行错误方案。

在正式进入开发之前，开发者/我和 AI Agent 必须先共同打好最稳固的 AI Agent 驱动基础设施，也就是先建立需求文档、架构文档、任务清单、时间切片、状态管理、完成判定、测试验收、失败修复、上下文防溢出、Agent 交接、skills 目录、工具使用规范、代码提交规范、文档更新规范等基础设施，然后开发者/我才逐渐退出具体开发过程。之后我基本不参与具体实现，只会不断发送“下一步”，其他事情都由 AI Agent 按照已经建立好的文档体系和任务清单来实现，包括规划、拆解、写代码、测试、修复、记录状态、更新文档、判断是否完成、推进下一个任务、生成交接说明、维护项目记忆等等。

这个开发过程不能一步到位，不能一上来就直接写完整项目，也不能跳过需求确认、架构设计、任务拆解和基础设施建设，必须先有项目的大致构图、整体流程路线、阶段路线图、功能模块拆分、子模块拆分、任务清单、时间切片、验收门槛、风险清单和状态追踪机制。每个大的流程都需要拆成多份时间切片，每个时间切片里面又做不同的工作，每个时间切片只能完成当前被分配的那一小块功能、流程、验证任务、文档任务或者修复任务，不能顺手扩展，不能临时优化别的东西，不能因为发现新想法就改变当前路线，不能在没有用户确认的情况下扩大需求范围。每完成一个时间切片或者一个功能模块的一部分，就必须进行明确状态标记，比如未开始、进行中、待验证、阻塞、需返工、已完成、已跳过、已归档等等，并且状态标记必须有证据支撑，不能只靠 AI 自己说完成了。

每个功能模块都要按照很庞大的工程模块来处理，不要把大型模块简化成几个小步骤，不要用乐观估算，而要使用最长时间估算、保守时间估算、工程级复杂度估算来分配任务。每个模块都必须拆成多个阶段、多个子模块、多个时间切片，每个时间切片都必须有明确输入、明确输出、允许修改的文件范围、禁止修改的文件范围、执行步骤、测试命令、验收标准、完成判定、失败处理方式和下一步进入条件。任务完成就是完成，没完成就是没完成，不能模糊处理，不能把“代码写了”当成“任务完成”，不能把“理论上可行”当成“已经完成”，不能把“还没运行但看起来没问题”当成“已完成”。

AI 必须有明确的完成判定标准，否则它可能会在代码还跑不通、依赖没装好、测试没通过、需求没对齐、文档没更新、状态没记录、上下文没沉淀的时候，就强行打上完成标记。所以每个任务必须明确什么情况下才算完成，什么情况下只能算待验证、阻塞或者需返工。完成标准至少要包括：代码已实现，代码能运行，关键路径测试通过，没有明显运行时错误，没有破坏已有功能，符合需求文档，符合架构文档，符合任务清单，相关文档已更新，状态文件已更新，变更记录已更新，问题记录已更新，下一步任务已明确，并且最好保留测试命令、运行结果、修复记录、验证结论等证据。如果其中任何一项没有满足，AI Agent 都不能标记为已完成，只能标记为待验证、阻塞或需返工。

整个开发过程必须避免死亡优化循环，不要无限重构、无限修复、无限优化、无限改架构，也不要因为发现新想法就偏离当前任务。必须完全按照任务清单开发，每个时间切片只做任务清单里定义的内容。如果遇到问题就按照失败处理流程进行修复，修复次数必须有限制，比如同一个问题最多尝试三轮修复，每一轮修复都要记录原因、修改内容、验证结果。如果三轮之后仍然失败，就必须生成问题报告、阻塞说明、可选解决方案和需要用户确认的决策点，而不是继续盲目修复。非关键优化、代码美化、架构升级、体验增强、新功能想法都不能打断当前主线任务，只能进入 backlog，除非它们直接影响运行、测试、需求一致性、安全性或核心功能验收。

这份 skill / prompt 必须能驱动 Codex、Claude Code、Kiro、Cursor、Google Anti 2 / Google Antigravity 等 AI 编程 Agent 使用，不能只适配某一个工具，必须说明不同 Agent 如何读取项目文档、如何理解当前状态、如何找到当前唯一合法任务、如何执行任务、如何运行测试、如何修复失败、如何更新状态、如何生成交接说明、如何等待下一步、如何在上下文丢失后恢复开发。整个项目不能依赖聊天上下文，因为上下文可能溢出、会话可能中断、工具可能切换、不同 Agent 的记忆能力也不同，所以必须要求 AI Agent 在项目里持续维护项目文档和状态文件。目录规则必须明确：面向开发者、二次开发者和维护者的人类项目文档统一放在 `docs/` 下；AI Agent 运行产物、状态文件、中间模板、任务拆分、测试报告、问题报告、风险登记、交接说明、学习记录、skill 定义和工具适配规则等全部放在 `docs/ai/*` 下，例如 `docs/ai/brief/PROJECT_BRIEF.md`、`docs/ai/requirements/REQUIREMENTS.md`、`docs/ai/architecture/ARCHITECTURE.md`、`docs/ai/planning/ROADMAP.md`、`docs/ai/tasks/TASKS.md`、`docs/ai/tasks/TIME_SLICES.md`、`docs/ai/state/STATUS.md`、`docs/ai/decisions/DECISIONS.md`、`docs/ai/changes/CHANGELOG.md`、`docs/ai/issues/ISSUES.md`、`docs/ai/acceptance/ACCEPTANCE.md`、`docs/ai/handoff/AGENT_HANDOFF.md`、`docs/ai/backlog/BACKLOG.md`、`docs/ai/reports/TEST_REPORT.md`、`docs/ai/risks/RISK_REGISTER.md`、`docs/ai/skills/SKILLS.md`、`docs/ai/rules/AGENT_RULES.md` 等等。根目录应尽量保持干净，只保留项目自身必要文件和极少数工具强制要求的入口文件；如果某个工具必须读取根目录规则文件，也应该让该入口文件指向 `docs/ai/rules/` 中的权威来源。每次开始任务前都要先读取这些文档，而不是依赖之前的聊天记录，每次完成任务后也必须更新这些文档，保证换一个 Agent、换一个工具、重启会话、上下文清空之后还能继续开发。

这份文档还必须要求 AI Agent 在项目开始阶段生成一套项目专用的 AI Agent skills，而不是只有一个总 prompt。这些 skills 必须放入 `docs/ai/skills/*/SKILL.md` 或等价的 `docs/ai/skills/` 子结构中，至少应该包括：需求澄清 skill，用来从用户原始想法中提炼目标和边界；需求生成 skill，用来产出可确认的需求文档；用户修正审查 skill，用来判断开发者提出的修正是否合理；架构拆解 skill，用来生成工程架构和模块边界；任务拆分 skill，用来把架构拆成任务清单；时间切片 skill，用来把任务拆成可执行的小块；代码执行 skill，用来按照任务清单写代码；测试验证 skill，用来运行测试并判断是否通过；失败修复 skill，用来限制修复次数并生成问题报告；状态更新 skill，用来维护 `docs/ai/state/STATUS.md`、`docs/ai/tasks/TASKS.md`、`docs/ai/tasks/TIME_SLICES.md` 等文件；上下文恢复 skill，用来在新会话中恢复当前进度；Agent 交接 skill，用来让 Codex、Claude Code、Kiro、Cursor、Google Antigravity 等不同 Agent 可以接手；防死亡优化 skill，用来阻止无限优化和偏离主线；验收判定 skill，用来防止代码没跑通就被标记完成。

这份 skill / prompt 还必须要求 AI Agent 具备可追溯的自主学习能力，但这种学习不能依赖隐藏记忆，也不能让 Agent 私下改变规则。AI Agent 必须从项目文档、代码、测试结果、失败记录、用户修正、验收反馈、外部文档引用和历史决策中总结可复用经验，并把学习结果沉淀到 `docs/ai/*` 下的仓库文件中，例如 `docs/ai/skills/SKILLS.md`、`docs/ai/skills/*/SKILL.md`、`docs/ai/decisions/DECISIONS.md`、`docs/ai/risks/RISK_REGISTER.md`、`docs/ai/reports/TEST_REPORT.md`、`docs/ai/handoff/AGENT_HANDOFF.md` 或 `docs/ai/learning/` 下的专门学习记录文件。每条学习都必须标注来源、证据、适用范围、风险、是否已由开发者确认、是否会改变后续 Agent 行为。AI Agent 不得把未经验证的猜测当成已确认经验，不得用自主学习绕过需求、架构、测试、权限或安全约束，不得因为学习到新想法就打断当前时间切片；新经验如果不影响当前验收，应进入 `docs/ai/backlog/` 或 `docs/ai/skills/` 下的 skill 候选区，等待合适时机处理。

这份 skill / prompt 还必须要求 AI Agent 为开发者、二次开发者、维护者和未来接手项目的人长期沉淀一套人类可读的项目文档，而不只是维护给 Agent 执行用的状态文件。这套长期项目文档必须统一放在 `docs/` 下，服务于扩展迭代、二次开发、学习理解、重构规划、问题排查、交接维护和长期演进；所有 AI 过程产物和 Agent 内部工作文件必须放在 `docs/ai/*` 下，不得混入 `docs/` 的人类长期文档层。AI Agent 必须区分“短期状态文件”和“长期知识文档”：短期状态文件用于判断当前唯一合法任务，长期知识文档用于让新人理解项目为什么这样设计、如何运行、如何扩展、哪些边界不能破坏、哪些技术债需要谨慎处理。长期文档至少应覆盖项目总览、业务目标、用户场景、核心概念、架构总览、模块说明、数据模型、接口边界、扩展点、开发环境、运行方式、测试策略、发布/部署说明、故障排查、重构指南、二次开发指南、学习路线、术语表、重要决策、已知限制、技术债和未来演进方向。AI Agent 每次完成关键变更后都必须判断是否需要更新这些长期文档，并记录文档更新证据；如果文档与代码不一致，必须在 `docs/ai/issues/` 或 `docs/ai/backlog/` 中标记为文档债或待修复问题，不能让过期文档继续误导开发者。文档必须分层组织，避免把所有内容堆进一个文件：面向新人快速理解的文档要短，面向维护者的细节可以放在 `docs/` 子目录；稳定知识和频繁变化状态必须分开；示例、教程、设计说明、故障手册和重构计划也应分开。AI Agent 不得为了“沉淀文档”制造无意义长文档，不得重复复制同一事实到多个地方；如果同一事实必须出现在多处，必须指定权威来源。

这份 skill / prompt 还必须要求 AI Agent 具备创建、更新、验证和废弃项目专用 skills 的能力。AI Agent 应该在发现重复任务、脆弱流程、领域知识、工具集成、测试/发布流程、错误修复套路或项目特有规范时，判断是否需要创建新的 skill；如果需要，必须先说明触发原因、适用场景、输入输出、边界、验证方式和与现有 skills 的关系，再在 `docs/ai/skills/` 下生成简洁的 `SKILL.md`。每个 skill 都必须包含清晰的名称、触发条件、用途、必读输入、执行步骤、产出、验收标准、失败处理和证据要求；如果内容过长，必须使用渐进披露，把详细参考资料放入该 skill 目录下的 `references/`，把可重复执行且要求稳定的逻辑放入该 skill 目录下的 `scripts/`，把输出资源放入该 skill 目录下的 `assets/`，避免把所有内容塞进一个超长 skill。AI Agent 更新 skill 时必须记录变更原因和兼容性影响；废弃 skill 时必须说明替代方案；创建或修改会显著改变工作流的 skill 前，必须请求开发者确认。

当开发者/我发送“下一步”的时候，AI Agent 不能随便选择任务，必须先读取项目状态文件，确认当前阶段、当前模块、当前时间切片、当前任务状态、阻塞项、上一次执行结果、下一步进入条件，然后只执行任务清单中当前唯一合法的下一项任务。如果没有明确的下一项任务，AI Agent 必须先生成状态诊断报告，而不是直接编造任务。如果当前任务处于阻塞状态，AI Agent 必须先处理阻塞或请求必要决策，而不是跳过。如果当前任务处于待验证状态，AI Agent 必须先完成验证，而不是继续写新代码。如果当前任务已经完成，AI Agent 才能推进到下一个时间切片。每次执行“下一步”后，AI Agent 都必须输出本次做了什么、修改了哪些文件、执行了哪些命令、测试结果是什么、当前任务是否完成、为什么这样判定、状态文件更新了什么、下一步应该做什么。

这份 skill / prompt 还必须包含清晰的版本策略和演进策略，而不是只给出一个静态提示词。你必须要求 AI Agent 在生成的文档中说明 prompt 版本号、适用的 Agent 和工具版本范围、变更类型、兼容性说明、弃用规则、迁移方式、CHANGELOG 记录方式、从原始需求到最终 prompt 的追溯方式，以及未来修改 prompt 时如何判断是否破坏了原始目标。版本策略不需要复杂到像软件库一样沉重，但必须足够让后续迭代知道哪些变更是修正、哪些变更是新增能力、哪些变更会改变 Agent 行为。

这份 skill / prompt 必须明确适用边界，说明它适合什么项目、不适合什么项目、什么时候应该降级为轻量流程、什么时候必须使用完整工程流程、什么时候必须停止并请求开发者决策。适用边界至少要区分：新项目启动、已有项目接手、单次 bug 修复、文档整理、原型验证、研究任务、生产级项目、个人小工具、团队协作项目、涉及安全或数据风险的项目。不要把所有项目都强行纳入同一套重型流程；AI Agent 必须能根据项目规模、风险、生命周期、协作人数、是否生产使用、是否涉及外部依赖和是否需要长期维护来选择合适的流程强度。

这份 skill / prompt 必须提供轻量模式、标准模式和严格模式三种工作模式。轻量模式适合一次性、小型、低风险任务，只维护最少状态文件和最短确认流程；标准模式适合普通工程项目，需要需求、架构、任务、状态、测试和交接；严格模式适合长期、多人、生产级或高风险项目，需要完整文档体系、风险登记、决策记录、CHANGELOG、测试报告、Agent skills、上下文恢复和交接机制。每种模式都必须说明进入条件、必需文件、可选文件、执行节奏、验收要求、升级到更严格模式的触发条件、降级到更轻量模式的条件，以及开发者未指定模式时的默认选择规则。

这份 skill / prompt 必须包含关键场景的示例输出，而不仅是抽象规则。示例输出至少要覆盖：首次项目启动时的回复示例、需求澄清模板示例、技术方案选项示例、用户提出不合理修正时的审查回复示例、开发者只发送“下一步”时的执行报告示例、状态不清楚时的状态诊断报告示例、测试失败后的失败修复记录示例、三轮修复失败后的阻塞报告示例、任务完成时的验收判定示例、Agent 交接说明示例。示例必须短而清晰，能让不同 AI Agent 模仿输出结构，但不能把示例当成固定内容机械复制。
