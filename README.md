# AI Agent 工程级 Prompt 食用教程

这个仓库的核心文件是 [`prompt.md`](./prompt.md)。它是一份给 AI 编程 Agent 食用的工程级工作协议，用来指导 Codex、Claude Code、Kiro、Cursor、Google Antigravity / Antigravity CLI 等工具，从一个模糊想法开始，逐步完成需求澄清、架构规划、任务拆分、执行切片、测试验收、状态记录和项目交接。

它不是普通开发教程，也不是“让 AI 直接写完整项目”的万能咒语。它的重点是：让 Agent 的代理行动权可确认、可审计、可暂停、可回滚、可收回。

## 适合谁用

适合这些场景：

- 你只有一个大概想法，不知道怎么表达成工程需求。
- 你希望 AI Agent 帮你从 0 到 1 推进项目。
- 你希望项目过程可追踪，而不是全靠聊天上下文。
- 你要接手已有项目，希望 AI 先看懂仓库再动手。
- 你要做生产、多用户、安全、权限、数据或长期维护相关项目。
- 你想在 Codex、Claude Code、Cursor、Kiro、Antigravity 等工具之间迁移同一套工作方式。

不适合：

- 只想让 AI 一次性输出一段小代码。
- 不希望记录状态、需求和验证证据。
- 希望 AI 无确认地操作生产环境、真实数据或高风险权限。

## 快速食用

1. 打开 [`prompt.md`](./prompt.md)。
2. 把全文作为系统提示词、项目规则、Agent rule、custom instruction 或项目 prompt 提供给你的 AI 编程工具。
3. 给 Agent 一个项目想法，例如：

```text
我想做一个本地工具，帮我把零散的产品想法整理成需求文档和任务清单。
```

4. Agent 应该先做项目类型判断、推荐轻量/标准/严格模式，并用几个选择题引导你补充需求。
5. 你确认需求和第一版范围后，Agent 才应该建立项目文档、任务树和执行切片。
6. 后续你可以发送：

```text
下一步
```

Agent 会读取项目状态文件，找到当前唯一合法执行切片，然后继续推进。

## 推荐启动话术

可以直接复制下面这段给 Agent：

```text
请按照本项目 prompt.md 的协议工作。先不要写代码。
请先根据我的想法判断项目类型、风险等级、推荐工作模式和当前代理行动权级别。
然后用普通人能理解的方式，引导我确认第一版需求范围。
我的想法是：……
```

如果是已有项目：

```text
请按照本项目 prompt.md 的协议接手这个已有仓库。
先只读扫描项目结构、README、测试、CI、依赖、已有文档和未提交改动。
不要修改文件，不要安装依赖，不要运行高风险命令。
扫描后告诉我项目类型、风险、推荐模式和下一步需要确认的事项。
```

## 三种工作模式

### 轻量模式

用于一次性、低风险、小范围任务。

保留最小需求记录、任务边界、当前状态、验证结果和变更记录。不要生成大量空壳文档。

### 标准模式

用于普通工程项目。

需要需求、架构、任务、执行切片、状态、测试、变更和交接记录。Agent 可以在已确认执行切片内受控推进。

### 严格模式

用于生产、多用户、安全、隐私、权限、资金、外部服务、合规或长期维护项目。

需要完整文档体系、风险登记、决策记录、测试报告、项目专用 skills、上下文恢复、多 Agent 交接和回滚方案。

## 核心概念

- `代理行动权`：你授权 Agent 做工程动作的范围。它不是无限自治，必须可暂停、可审计、可回滚。
- `需求台账`：记录所有需求输入、来源、状态和去向，避免需求只存在聊天里。
- `定稿需求`：你确认后的当前版本需求基线。
- `任务树`：项目目标、阶段、模块、任务、执行切片、执行步骤的层级。
- `执行切片`：Agent 单次推进的最小可验证闭环。每次“下一步”都应该只执行一个合法切片。
- `完成判定`：代码写完不等于完成。完成还需要验证、文档、状态、变更和授权合规。

## 建议目录

`prompt.md` 要求项目文档分两层：

```text
docs/
  面向开发者、二次开发者和维护者的长期文档
  ai/
    AI Agent 的需求、任务、状态、测试、风险、交接、skills 等过程产物
```

常见 AI 文件包括：

```text
docs/ai/requirements/REQUIREMENT_LEDGER.md
docs/ai/requirements/REQUIREMENTS.md
docs/ai/tasks/TASKS.md
docs/ai/tasks/EXECUTION_SLICES.md
docs/ai/state/STATUS.md
docs/ai/reports/TEST_REPORT.md
docs/ai/changes/CHANGELOG.md
docs/ai/handoff/AGENT_HANDOFF.md
```

轻量模式可以简化，不需要一开始生成完整目录。

## “下一步”怎么用

当你发送“下一步”时，Agent 不应该随便挑任务。它必须先读取状态文件，按优先级处理：

1. 阻塞项和必要决策点。
2. 待验证项。
3. 需返工项。
4. 当前进行中的执行切片。
5. 满足条件的下一个未开始执行切片。

如果无法确定唯一切片，Agent 必须生成状态诊断报告，而不是编造任务。

## 安全边界

这些事情必须暂停并请求你确认：

- 删除数据、批量迁移、数据库变更。
- 生产部署、真实账号、外部服务账单。
- 密钥、隐私数据、权限、认证、支付。
- 安装不可信依赖或改变 lockfile 带来供应链风险。
- 修改架构、技术栈、数据模型、模块边界。
- 改变需求范围、P0 功能或明确不做的内容。

Agent 应优先提出低风险替代方案，例如只读扫描、先备份、测试环境验证、dry-run、生成迁移计划。

## 修改 prompt.md 时

后续修改 [`prompt.md`](./prompt.md) 时，请保留这些能力：

- 普通想法者引导。
- 项目类型判断。
- 轻量/标准/严格模式裁剪。
- 需求归集与需求台账。
- 定稿需求确认。
- 用户修正审查。
- 架构规划。
- 任务树和执行切片。
- 状态管理和“下一步”协议。
- 测试验收和完成判定。
- 三轮失败修复上限。
- 文档沉淀和上下文恢复。
- 项目专用 skills。
- 多工具适配。
- 安全与外部输入防护。
- 代理行动权治理。

如果修改会改变确认边界、完成判定、目录结构、安全规则、“下一步”协议或 Agent 默认行为，应当升级版本并记录兼容性影响。

## 一句话记忆

这份 prompt 的目标不是让 AI 更“自由”，而是让 AI 在工程项目里更可靠：知道该问什么、能做什么、做到哪里必须停、怎么留下证据，以及未来怎么接着做。
